CLUB NOTIFICATIONS SPEC (BACKEND READY)
======================================

Source scope:
- Club notification center page
- Club proposal flow
- Membership flow
- Vacancy flow
- Events flow
- Club admin pages


1) NOTIFICATIONS EXPLICITLY SHOWN IN CLUB NOTIFICATION PAGE
-----------------------------------------------------------

1. Club proposal approved
- Sender: Student Life Office / Clubs Admin
- Receiver: Club proposal owner (student who submitted proposal)
- Trigger: Proposal status changed to Approved
- For Student Services review flow, exact title/body copy and broadcast variant: see §2 items 21–22.
- Required details:
  - proposalId
  - clubName
  - decision: approved
  - reviewedBy (admin/office id)
  - reviewedAt
  - nextActionUrl: /club-admin
  - optional notes

2. Club proposal revision required
- Sender: Student Life Office / Clubs Admin
- Receiver: Club proposal owner
- Trigger: Proposal status changed to Revision Required
- For Student Services “request revision” notification copy: see §2 item 24.
- Required details:
  - proposalId
  - clubName
  - decision: revision_required
  - reviewComments[] (list)
  - reviewedBy
  - reviewedAt
  - resubmitUrl: /clubs/propose

3. Membership application approved
- Sender: Club executive team / club admin
- Receiver: Student applicant
- Trigger: Membership application status changed to Approved
- Required details:
  - membershipApplicationId
  - clubId
  - clubName
  - decision: approved
  - roleAssigned (optional)
  - effectiveFrom
  - approvedBy
  - approvedAt
  - clubUrl: /clubs/my-memberships

4. Vacancy interview invitation
- Sender: Club recruiter / club admin
- Receiver: Vacancy applicant
- Trigger: Vacancy application moved to interview stage
- Required details:
  - vacancyApplicationId
  - vacancyId
  - clubId
  - clubName
  - positionTitle
  - interviewSlots[] or proposedSlot
  - respondBy
  - meetingMode
  - meetingLink/location
  - actionUrl: /clubs/vacancies/my-applications

5. Event reminder
- Sender: Event system (or club event organizer)
- Receiver: Registered attendee
- Trigger: Reminder schedule hit (example: 2h before event)
- Required details:
  - eventId
  - eventTitle
  - clubId
  - clubName
  - startAt
  - location
  - ticketId (optional)
  - reminderOffset (example: 2h_before)
  - registrationsUrl: /clubs/events/my-registrations

--- Club proposals (Student Services review) — same notification structure as event proposals (17–19) ---

These map to `POST /api/student-services/club-proposals/:proposalId/approve`, `.../reject`, and `.../request-revision` (see club-student-services-endpoints.md). Use the common notification object in §3 (title, body, receiver, data, actionUrl, etc.).

21. Club proposal approved — to proposer
- Sender: Student Services / Student Life Office
- Receiver: Proposer (student who submitted the proposal)
- Trigger: Club proposal approved via Student Services
- Notification display (in-app / email):
  - title: "Your Club Proposal is approved"
  - body: "Click to launch club Dashboard"
- Required details (payload):
  - proposalId
  - clubId (when provisioned)
  - clubName
  - decision: approved
  - reviewedBy
  - reviewedAt
  - actionUrl: /club-admin (club dashboard launch)
- Suggested type / routing key: student_services.club_proposal.approved.proposer (or club.proposal.approved.proposer under §5)

22. Club proposal approved — broadcast to all users
- Sender: Student Services / System
- Receiver: All users (broadcast segment; not a single userId)
- Trigger: Same approval event as (21); emitted in addition to the proposer notification
- Notification display:
  - title: "New Club Alert"
  - body: {clubName} (the approved club name as plain text)
- Required details:
  - proposalId
  - clubId
  - clubName
  - announcedAt
  - audience: all_users (or segment id used by product)
  - deepLink: optional /clubs/{clubId} or equivalent
- Suggested type / routing key: student_services.club_proposal.approved.broadcast

23. Club proposal disapproved — to proposer
- Sender: Student Services / Student Life Office
- Receiver: Proposer
- Trigger: Club proposal rejected via Student Services (`POST .../reject` with reason)
- Notification display:
  - title: "Your Club Proposal is Disapproved"
  - body: {adminNote} (same text as rejection reason / admin note to the proposer)
- Required details:
  - proposalId
  - clubName (draft name from proposal)
  - decision: rejected
  - adminNote (maps to API `reason` field)
  - reviewedBy
  - reviewedAt
  - optional resubmitUrl: /clubs/propose
- Suggested type / routing key: student_services.club_proposal.rejected

24. Club proposal — request revision (Request revision pressed)
- Sender: Student Services / Student Life Office
- Receiver: Proposer
- Trigger: `POST .../club-proposals/:proposalId/request-revision` (comments to proposer)
- Notification display:
  - title: "There is a need for Changes in your Club Proposal"
  - body: "Go to Announcements on Web to see comments and modify"
- Required details:
  - proposalId
  - clubName
  - decision: revision_required
  - changes / reviewComments (from API `changes` or threaded comments)
  - reviewedBy
  - reviewedAt
  - announcementsUrl or deepLink to Student Services / web announcements surface where comments appear
- Suggested type / routing key: student_services.club_proposal.revision_requested

25. New club announcement broadcast

3) RECOMMENDED COMMON NOTIFICATION OBJECT
-----------------------------------------

Use one common schema for all club notification types:

- id
- type (examples: club.proposal.approved, vacancy.application.rejected)
- sender:
  - actorType
  - actorId
  - actorName
- receiver:
  - userId
  OR receivers[] for broadcast notifications
- title
- body
- entity:
  - entityType
  - entityId
- data (type-specific payload)
- channels[] (in_app, email, optional push)
- priority (low|normal|high)
- status (unread|read|archived)
- createdAt
- readAt
- actionUrl


4) RABBITMQ QUEUE NAME ALTERNATIVES
-----------------------------------

Goal:
- Keep naming consistent, environment-aware, and easy to route/monitor.

General recommendations:
- Use lowercase with dots: domain.feature.action
- Add environment prefix for shared brokers: dev., stg., prod.
- Separate command vs event queues if needed
- Keep dead-letter queues explicit with .dlq suffix

Alternative A: One consolidated club notifications queue
- Queue:
  - notifications.club
- Environment variants:
  - dev.notifications.club
  - stg.notifications.club
  - prod.notifications.club
- DLQ:
  - notifications.club.dlq

Alternative B: By channel (in-app / email / push)
- Queues:
  - notifications.club.inapp
  - notifications.club.email
  - notifications.club.push
- Environment variants:
  - dev.notifications.club.inapp
  - dev.notifications.club.email
  - dev.notifications.club.push
- DLQs:
  - notifications.club.inapp.dlq
  - notifications.club.email.dlq
  - notifications.club.push.dlq

Alternative C: By business domain in clubs module
- Queues:
  - notifications.club.proposals
  - notifications.club.membership
  - notifications.club.vacancies
  - notifications.club.events
  - notifications.club.announcements
- Example env variants:
  - prod.notifications.club.proposals
  - prod.notifications.club.membership
  - prod.notifications.club.vacancies
  - prod.notifications.club.events
  - prod.notifications.club.announcements
- DLQ pattern:
  - {queue}.dlq

Alternative D: By priority/SLA
- Queues:
  - notifications.club.high
  - notifications.club.normal
  - notifications.club.low
- Good for:
  - high: interview invites, approval/rejection decisions
  - normal: confirmations, submissions
  - low: reminders, informational updates
- DLQs:
  - notifications.club.high.dlq
  - notifications.club.normal.dlq
  - notifications.club.low.dlq

Alternative E: Event-driven naming (topic style)
- Queues (consumer-specific):
  - notifications.club.proposal-events
  - notifications.club.membership-events
  - notifications.club.vacancy-events
  - notifications.club.event-events
- Typical routing keys:
  - club.proposal.approved
  - club.proposal.revision_required
  - club.membership.approved
  - club.membership.rejected
  - club.vacancy.application.submitted
  - club.vacancy.interview.invited
  - club.event.reminder
  - club.announcement.created

Suggested practical default (balanced):
- Exchange:
  - notifications.club.topic
- Main queues:
  - notifications.club.inapp
  - notifications.club.email
- Optional:
  - notifications.club.push
- DLQs:
  - notifications.club.inapp.dlq
  - notifications.club.email.dlq
  - notifications.club.push.dlq
- Retry queues (optional):
  - notifications.club.inapp.retry
  - notifications.club.email.retry
  - notifications.club.push.retry

Mapping examples (type -> routing key):
- club.proposal.approved -> club.proposal.approved
- club.proposal.revision_required -> club.proposal.revision_required
- club.membership.approved -> club.membership.approved
- club.membership.rejected -> club.membership.rejected

- club.vacancy.application.approved -> club.vacancy.application.approved

5) FINAL SELECTED CONVENTION (IMPLEMENTATION READY)
----------------------------------------------------

Use this convention as the default in backend implementation.

Environment:
- Prefix all broker artifacts with environment:
  - dev
  - stg
  - prod

Exchange:
- {env}.notifications.club.topic
- Type: topic
- Durable: true

Primary queues:
- {env}.notifications.club.inapp
- {env}.notifications.club.email
- {env}.notifications.club.push (optional if push channel is enabled)

Dead-letter queues:
- {env}.notifications.club.inapp.dlq
- {env}.notifications.club.email.dlq
- {env}.notifications.club.push.dlq

Retry queues (delayed re-processing):
- {env}.notifications.club.inapp.retry
- {env}.notifications.club.email.retry
- {env}.notifications.club.push.retry

Routing key convention:
- club.<domain>.<action>[.<subaction>]
- Examples:
  - club.proposal.approved
  - club.proposal.revision_required
  - club.membership.submitted
  - club.membership.approved
  - club.membership.rejected
  - club.vacancy.application.submitted
  - club.vacancy.application.approved
  - club.vacancy.application.rejected
  - club.vacancy.interview.invited
  - club.event.registration.confirmed
  - club.event.registration.cancelled
  - club.event.reminder
  - club.announcement.created

Queue bindings (recommended):
- {env}.notifications.club.inapp binds:
  - club.#
- {env}.notifications.club.email binds:
  - club.proposal.#
  - club.membership.#
  - club.vacancy.#
  - club.event.registration.#
  - club.event.reminder
  - club.announcement.created
- {env}.notifications.club.push binds (optional):
  - club.proposal.approved
  - club.membership.approved
  - club.vacancy.interview.invited
  - club.event.reminder

Message headers (recommended):
- x-notification-id
- x-notification-type
- x-channel (in_app|email|push)
- x-priority (high|normal|low)
- x-correlation-id
- x-user-id (single receiver) OR x-audience-id (broadcast segment)

Operational defaults:
- Delivery mode: persistent
- Queue durability: true
- Ack mode: manual ack
- Retry policy:
  - 1st retry: 30s
  - 2nd retry: 5m
  - 3rd retry: 30m
  - then route to DLQ
